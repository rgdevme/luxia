# Documentation authoring (with change log)

Follow these conventions whenever you create or edit documentation under the
project's docs root (the `docs.root` configured in `agnos.json`, default `.docs`).

## Frontmatter

Every concept document must begin with YAML frontmatter carrying this exact
shape. `type`, `title`, `description`, and `timestamp` must be present and
non-empty. `resource` and `tags` must be present but may be empty (`""` and `[]`).

```markdown
---
type: Concept kind, e.g. "Technical Doc", "Playbook", "BigQuery Table".
title: Short, human-readable title of the document.
description: One- or two-sentence summary of what the document covers.
resource: URI uniquely identifying the underlying asset. Use "" if none.
tags: [] # List of categorization strings. Use [] if none.
timestamp: 2026-06-30T00:00:00Z # ISO 8601 datetime of the last change.
---
```

Update `timestamp` to the current time whenever you change a document.
Producers may add custom fields; they are tolerated and ignored by the index.

## Body sections

Use these conventional headings when their content applies (OKF):

- `# Schema` for a structured description of an asset's columns or fields.
- `# Examples` for concrete usage examples.
- `# Citations` for external sources, numbered and linked.

## Reserved files

- `index.md` is autogenerated by `agnos docs`. Never edit it by hand.
- `log.md` is a reserved change history. Do not list it as a concept document.

## Writing conventions

- Do not use em dashes.
- Do not sign documentation as yourself.
- Do not write token values, code snippets, or anything already stated in code,
  unless it is a reference.
- Do not duplicate values that live elsewhere. Reference the file path instead.

## Linking conventions

- All cross-references between files must use Markdown links:
  `[display text](relative/path.md)`. Bundle-relative absolute links begin with
  `/` and resolve from the bundle root; relative links resolve normally.
- Do not use bare backtick-quoted paths for internal links. Backtick strings are
  acceptable only for references.

## Documenting conventions

- If a task changes how a system works (auth flow, data model, API pattern,
  subscription logic, etc.), update the relevant file in `<docs-root>/technical/`.
- If a task introduces an architectural decision (a new pattern, library, or
  architectural approach), document it in `<docs-root>/technical-decisions`.
- Do not update business, product, or design docs automatically. Surface gaps
  and let the user decide.
- If a task implies an undocumented decision in any of these areas, flag it at
  the end of your response:
  > Doc gap detected: [Decision summary]. Consider updating
  > `<docs-root>/[section]/[file].md`.

## Maintain the log

After every documentation change, record it in `<docs-root>/log.md` as prose
(OKF log entries are prose, not generated output):

- Group entries under an ISO 8601 date heading (`## 2026-06-30`), newest first.
- Add one entry per change describing what changed and why, linking the affected
  documents with Markdown links.
- Append to today's heading if it already exists rather than creating a duplicate.
- Do not list `log.md` in the index; it is a reserved file.

Example:

```markdown
# Log

## 2026-06-30

- Documented the new auth flow in [Auth](technical/auth.md) after switching to
  token rotation.
- Removed the legacy session doc; the pattern no longer exists.
```
